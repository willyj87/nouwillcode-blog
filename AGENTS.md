## Technical Description

The stack is nextjs project with typescript, tailwindcss, shadcn and Sanity CMS.
Sanity CMS is used for content management, while Next.js serves as the frontend framework. Tailwind CSS provides utility-first styling, and Shadcn is used for building UI components. The project is structured to facilitate easy content updates and a responsive user interface.

## Agent Rules

You MUST adhere to the following rules when writing or modifying code in this project:

1. **Language (English only вЂ” no exceptions):** English MUST be used for absolutely everything in this project:
   - All user-facing / display text (UI labels, buttons, headings, placeholders, toast and error messages, empty states).
   - All Sanity schema strings (`title`, `description`, `initialValue`, `preview` titles/subtitles) and Studio structure labels.
   - All code comments and JSDoc.
   - All identifiers (variable, function, component, and file names).
   - All commit messages, PR titles/descriptions, and any documentation.
   - Date/number formatting must use an English locale (e.g. `date-fns` `enUS`), not `fr`.
   - The root `<html lang>` attribute must be `"en"`.
   Never introduce French (or any non-English) text. If you find existing non-English text, translate it to English as part of your change.
2. **React/Next.js Architecture:**
   - Use React Server Components (RSC) by default.
   - Only add the `"use client"` directive when strictly necessary for interactive components (e.g., using state, effects, or event listeners).
   - Use functional components and explicitly type your `Props` with TypeScript.
   - Standardize imports in the following order: `react` -> `next` -> external packages -> internal components -> `lib`/`utils`.
3. **Styling and UI (Tailwind & Shadcn):**
   - Systematically use the `cn()` utility (which combines `clsx` and `tailwind-merge`) provided by Shadcn to handle conditional Tailwind classes.
   - When you need a new base UI component, always generate it using the Shadcn CLI (e.g., `npx shadcn@latest add <component>`) instead of building it from scratch.
4. **Data Management (Sanity & GROQ):**
   - Centralize all GROQ queries in dedicated files (e.g., `src/sanity/lib/queries.ts`) rather than scattering them inline within components.
   - Strictly type the responses of your Sanity queries using the types automatically generated by Sanity TypeGen.
5. **Design Context (mandatory for UI/content work):**
   - Before changing any UI, layout, theme, typography, or UX copy, read and follow:
     - `PRODUCT.md` (strategic intent and brand principles)
     - `DESIGN.md` (visual system and implementation constraints)
   - Treat these two files as the source of truth for design decisions. If a new idea conflicts with them, prefer consistency with the documented system unless explicitly asked to evolve it.
6. **Core Product Principles (non-negotiable):**
   1. **Reading is the product.** Protect legibility, hierarchy, measure (65вЂ“75ch), and focus before adding ornament.
   2. **Authored, not templated.** Avoid generic, interchangeable blog scaffolds; preserve a distinct point of view.
   3. **Opinion earns attention.** Use decisive visual hierarchy and contrast; avoid timid, gray-on-gray design.
   4. **Credibility in the details.** States, spacing, motion, and code presentation must feel precise and intentional.
   5. **Fast and frictionless.** Prefer lightweight server-rendered experiences and avoid unnecessary client/runtime complexity.
7. **UI guardrails from the design system:**
   - Keep contrast at WCAG 2.1 AA minimum (body text в‰Ґ 4.5:1 in both themes).
   - Keep the palette neutral-first with indigo as a deliberate signal color (actions, links, focus), not decorative noise.
   - Avoid known anti-patterns: side-stripe accent borders, gradient text, glassmorphism-by-default, generic hero-metric templates, repetitive generic card grids, and reflexive uppercase eyebrow labels on every section.

## Project Goal

This project is about building a blog website with a content management system (CMS) using Sanity. The structure is designed to separate concerns, making it easier to manage content, UI components, and application logic.
